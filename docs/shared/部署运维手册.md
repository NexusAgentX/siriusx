## 环境变量清单

### 必填变量

| 变量名 | 说明 | 示例 |
|---|---|---|
| `NODE_ENV` | 运行环境 | `production` |
| `PORT` | HTTP 端口 | `3000` |
| `DATABASE_URL` | Postgres 连接串 | `postgres://user:pass@host:5432/db` |
| `REDIS_URL` | Redis 连接串 | `redis://host:6379` |
| `SESSION_SECRET` | Session 密钥（32+ 字符） | `your-secure-random-string-here` |
| `S3_BUCKET` | S3 存储桶名称（阶段 0 必需） | - |
| `S3_REGION` | S3 区域 | `us-east-1` |
| `AWS_ACCESS_KEY_ID` | 控制面访问 S3 的 AWS 访问密钥 | - |
| `AWS_SECRET_ACCESS_KEY` | 控制面访问 S3 的 AWS 密钥 | - |

**🚨 阶段 0 说明**：
- `S3_BUCKET` 及 AWS 凭证是阶段 0 **必需配置**
- 多 Control Worker 节点部署必须使用 S3 作为共享 workspace 存储
- 单节点开发环境可以使用本地 workspace（通过 `SIRIUSX_WORKSPACES_DIR`）
- 沙箱运行时不应直接配置长期 AWS 凭证，只接受控制面发放的短期 capability token

### 可选变量

| 变量名 | 说明 | 默认值 |
|---|---|---|
| `S3_ENDPOINT` | S3 兼容端点（MinIO 等） | - |
| `PI_AUTH_REGISTRY_PATH` | Pi SDK 认证存储路径 | `${SIRIUSX_DATA_DIR}/pi/auth` |
| `PI_MODEL_REGISTRY_PATH` | Pi SDK 模型注册表路径 | `${SIRIUSX_DATA_DIR}/pi/models` |
| `SIRIUSX_DATA_DIR` | 数据目录 | `/data/siriusx` |
| `SIRIUSX_WORKSPACES_DIR` | Workspace 目录（单节点开发用） | `${SIRIUSX_DATA_DIR}/workspaces` |
| `SIRIUSX_AGENT_CATALOG_DIR` | Agent 缓存目录 | `${SIRIUSX_DATA_DIR}/agent-catalog` |
| `SIRIUSX_SANDBOX_IMAGE` | Sandbox 镜像 | `siriusx/sandbox:latest` |
| `SANDBOX_RUNTIME_URL` | 沙箱运行时地址 | - |
| `SANDBOX_RUNTIME_TOKEN` | 控制面调用沙箱运行时的短期令牌 | - |
| `LOG_LEVEL` | 日志级别 | `info` |
| `MAX_WORKERS` | 最大 Control Worker 并发数 | `4` |
| `RUN_LEASE_TIMEOUT_MS` | Run lease 超时 | `90000` (90 秒) |
| `HEARTBEAT_INTERVAL_MS` | 心跳间隔 | `30000` (30 秒) |

## Docker Compose 部署

**🚨 部署模式说明**：

- **单体部署（开发/小规模）**：`runtime` 进程同时运行 API 和 Control Worker，简化部署
- **分布式部署（生产/大规模）**：独立的 `api` 和 `worker` 服务，可独立扩展；建议再拆出 `sandbox-runtime`

### 单体部署配置

**文件**：`docker-compose.yml`（开发/小规模生产）

适用场景：开发环境、MVP、小规模生产（< 50 并发 task）

```yaml
version: '3.8'

services:
  postgres:
    image: postgres:16
    restart: always
    environment:
      POSTGRES_DB: ${POSTGRES_DB}
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres-data:/var/lib/postgresql/data
    ports:
      - "127.0.0.1:5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    restart: always
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    volumes:
      - redis-data:/data
    ports:
      - "127.0.0.1:6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "--raw", "incr", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5

  runtime:
    image: siriusx/agent-runtime:${VERSION}
    restart: always
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: bun run start  # API + Control Worker 同进程
    environment:
      NODE_ENV: production
      PORT: 3000
      DATABASE_URL: postgres://${POSTGRES_USER}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB}
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379
      SESSION_SECRET: ${SESSION_SECRET}
      S3_BUCKET: ${S3_BUCKET}
      S3_REGION: ${S3_REGION}
      AWS_ACCESS_KEY_ID: ${AWS_ACCESS_KEY_ID}
      AWS_SECRET_ACCESS_KEY: ${AWS_SECRET_ACCESS_KEY}
      SIRIUSX_DATA_DIR: /data
      LOG_LEVEL: info
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - siriusx-data:/data
    ports:
      - "3000:3000"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  frontend:
    image: siriusx/agent-frontend:${VERSION}
    restart: always
    depends_on:
      - runtime
    environment:
      NODE_ENV: production
      PORT: 3456
      NEXT_PUBLIC_API_BASE: http://runtime:3000
    ports:
      - "3456:3456"

volumes:
  postgres-data:
  redis-data:
  siriusx-data:
```

### 分布式部署配置

**文件**：`docker-compose.production.yml`（生产环境推荐）

适用场景：生产环境、大规模部署（50+ 并发 task）、需要独立扩展 API、Control Worker 和 sandbox runtime

**架构目标**：2 API 节点 + 2 Control Worker 节点 + sandbox runtime（最小可行原型）

```yaml
version: '3.8'

services:
  postgres:
    image: postgres:16
    restart: always
    environment:
      POSTGRES_DB: ${POSTGRES_DB}
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./schema.sql:/docker-entrypoint-initdb.d/schema.sql
    ports:
      - "127.0.0.1:5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5
    deploy:
      resources:
        limits:
          cpus: '2'
          memory: 4G

  redis:
    image: redis:7-alpine
    restart: always
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    volumes:
      - redis-data:/data
    ports:
      - "127.0.0.1:6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "--raw", "incr", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5
    deploy:
      resources:
        limits:
          cpus: '1'
          memory: 2G

  api:
    image: siriusx/agent-runtime:${VERSION}
    restart: always
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: bun run start:api  # 只运行 API 服务
    environment:
      NODE_ENV: production
      PORT: 3000
      DATABASE_URL: postgres://${POSTGRES_USER}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB}
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379
      SESSION_SECRET: ${SESSION_SECRET}
      S3_BUCKET: ${S3_BUCKET}
      S3_REGION: ${S3_REGION}
      AWS_ACCESS_KEY_ID: ${AWS_ACCESS_KEY_ID}
      AWS_SECRET_ACCESS_KEY: ${AWS_SECRET_ACCESS_KEY}
      LOG_LEVEL: info
    ports:
      - "3000:3000"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    deploy:
      replicas: 2
      resources:
        limits:
          cpus: '2'
          memory: 4G
        reservations:
          cpus: '1'
          memory: 2G

  worker:
    image: siriusx/agent-runtime:${VERSION}
    restart: always
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    command: bun run start:worker  # 只运行 Control Worker 服务
    environment:
      NODE_ENV: production
      DATABASE_URL: postgres://${POSTGRES_USER}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB}
      REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379
      S3_BUCKET: ${S3_BUCKET}
      S3_REGION: ${S3_REGION}
      AWS_ACCESS_KEY_ID: ${AWS_ACCESS_KEY_ID}
      AWS_SECRET_ACCESS_KEY: ${AWS_SECRET_ACCESS_KEY}
      SIRIUSX_DATA_DIR: /data
      MAX_WORKERS: 4
      RUN_LEASE_TIMEOUT_MS: 90000
      HEARTBEAT_INTERVAL_MS: 30000
      LOG_LEVEL: info
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - siriusx-data:/data
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3001/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    deploy:
      replicas: 2
      resources:
        limits:
          cpus: '4'
          memory: 8G
        reservations:
          cpus: '2'
          memory: 4G

  frontend:
    image: siriusx/agent-frontend:${VERSION}
    restart: always
    depends_on:
      - api
    environment:
      NODE_ENV: production
      PORT: 3456
      NEXT_PUBLIC_API_BASE: https://api.siriusx.com
    ports:
      - "3456:3456"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3456"]
      interval: 30s
      timeout: 10s
      retries: 3
    deploy:
      replicas: 2
      resources:
        limits:
          cpus: '1'
          memory: 2G

volumes:
  postgres-data:
  redis-data:
  siriusx-data:
```

### 日志级别

| 级别 | 使用场景 |
|---|---|
| `error` | 错误和异常 |
| `warn` | 警告（如 lease 过期） |
| `info` | 关键业务事件（Task 创建、Run 完成） |
| `debug` | 调试信息 |
| `trace` | 详细追踪（仅开发环境） |

### 日志脱敏

```typescript
// 自动脱敏敏感字段
const SENSITIVE_FIELDS = ['password', 'token', 'secret', 'apiKey'];

function sanitizeLog(obj: any): any {
  if (typeof obj !== 'object' || obj === null) return obj;
  
  const sanitized = { ...obj };
  
  for (const key of Object.keys(sanitized)) {
    if (SENSITIVE_FIELDS.some(field => key.toLowerCase().includes(field))) {
      sanitized[key] = '***REDACTED***';
    } else if (typeof sanitized[key] === 'object') {
      sanitized[key] = sanitizeLog(sanitized[key]);
    }
  }
  
  return sanitized;
}
```

## 备份策略

### Postgres 备份

#### 自动备份脚本

```bash
#!/bin/bash
# backup-postgres.sh

BACKUP_DIR="/backups/postgres"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/siriusx_${TIMESTAMP}.sql.gz"

# 创建备份
pg_dump -h localhost -U siriusx -d siriusx | gzip > "${BACKUP_FILE}"

# 保留最近 7 天的备份
find "${BACKUP_DIR}" -name "*.sql.gz" -mtime +7 -delete

# 上传到 S3
aws s3 cp "${BACKUP_FILE}" s3://backups/postgres/
```

# 每天凌晨 2 点备份

## 故障排查

### 常见问题

#### 1. Session 无法持久化

**症状**：用户频繁掉线

**检查**：

```bash
# 检查 Redis 连接
redis-cli -h localhost -p 6379 ping

# 检查 session 存储
redis-cli -h localhost -p 6379 keys "sess:*"

# 查看 session 内容
redis-cli -h localhost -p 6379 get "sess:abc123"
```

**解决**：
1. 检查 `REDIS_URL` 配置
2. 检查 `SESSION_SECRET` 是否一致
3. 检查 Redis 内存是否充足

---

#### 2. Control Worker 无法认领 Run

**症状**：Run 一直处于 `queued` 状态

**检查**：

```bash
# 查看 Control Worker 日志
docker logs -f siriusx-runtime-1

# 检查 Redis 队列
redis-cli -h localhost -p 6379 llen "queue:tasks"

# 检查数据库 lease
psql -U siriusx -c "SELECT * FROM task_runs WHERE status = 'running' AND lease_expires_at < EXTRACT(EPOCH FROM NOW()) * 1000;"
```

**解决**：
1. 检查 Control Worker 进程是否正常运行
2. 检查 `MAX_WORKERS` 配置
3. 手动释放过期 lease：

```sql
UPDATE task_runs
SET status = 'queued', lease_owner_node_id = NULL, lease_expires_at = NULL
WHERE status = 'running' AND lease_expires_at < EXTRACT(EPOCH FROM NOW()) * 1000;
```

---

#### 3. Sandbox 创建失败

**症状**：Run 失败，错误信息 `Sandbox creation failed`

**检查**：

```bash
# 检查 Docker 守护进程
docker info

# 检查 Docker socket 权限
ls -la /var/run/docker.sock

# 检查 Sandbox 镜像
docker images | grep siriusx/sandbox

# 检查容器数量
docker ps -a | wc -l
```

**解决**：
1. 确保 Docker 守护进程运行
2. 确保 Runtime 有权限访问 Docker socket
3. 拉取 Sandbox 镜像：`docker pull siriusx/sandbox:latest`
4. 清理 orphan 容器：

```bash
docker rm -f $(docker ps -aq --filter "label=siriusx.managed=true")
```

---

#### 4. Workspace 冲突频繁

**症状**：大量 `WORKSPACE_CONFLICT` 错误

**检查**：

```sql
-- 查看最近的冲突
SELECT 
  task_id,
  COUNT(*) as conflict_count
FROM task_runs
WHERE status = 'failed' AND error LIKE '%WORKSPACE_CONFLICT%'
  AND created_at > EXTRACT(EPOCH FROM NOW() - INTERVAL '1 hour') * 1000
GROUP BY task_id
ORDER BY conflict_count DESC
LIMIT 10;
```

**解决**：
1. 检查是否有并发发送消息
2. 调整前端防止重复提交
3. 考虑增加排队机制

## 扩容指南

### 水平扩容

#### API Runtime

```yaml
# docker-compose.yml
runtime:
  deploy:
    replicas: 4  # 增加副本数
```

**注意事项**：
- 需要配置负载均衡（如 Nginx）
- Session 必须使用 Redis 存储
- 文件上传需要使用共享存储

#### Control Worker Runtime

```yaml
worker:
  deploy:
    replicas: 8  # 增加 Control Worker 数量
  environment:
    MAX_WORKERS: 4  # 每个进程的并发数
```

**容量规划**：
- 1 个 Control Worker 进程：4 并发 run 编排
- 1 个 Control Worker 容器：2-4 个进程
- 100 并发 tasks：需要 6-8 个 Control Worker 容器

#### Sandbox Runtime

```yaml
sandbox-runtime:
  deploy:
    replicas: 8  # 增加一次性沙箱执行能力
```

**注意事项**：
- 沙箱运行时只接收 `RunSpec` 和短期 capability token
- 不配置数据库、S3 或用户凭证主密钥
- 按 CPU、内存、浏览器并发和 sandbox 创建耗时独立扩容

### 垂直扩容

#### 增加 Postgres 资源

```yaml
postgres:
  deploy:
    resources:
      limits:
        cpus: '4'      # 增加 CPU
        memory: 8G     # 增加内存
```

#### 优化查询性能

```sql
-- 开启 pg_stat_statements
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 查看慢查询
SELECT 
  query,
  calls,
  total_time,
  mean_time
FROM pg_stat_statements
ORDER BY mean_time DESC
LIMIT 10;
```

## 安全加固

### 网络隔离

```yaml
# docker-compose.yml
networks:
  frontend:
    internal: false  # 允许外部访问
  backend:
    internal: true   # 内部网络

services:
  frontend:
    networks:
      - frontend
  
  runtime:
    networks:
      - frontend
      - backend
  
  postgres:
    networks:
      - backend  # 不暴露给外部
```

### 版本升级

1. **备份数据库**
2. **运行迁移脚本**：

```bash
bun run migrate:up
```

3. **滚动更新服务**：

```bash
docker-compose up -d --no-deps --build runtime
```

4. **验证功能**
5. **回滚准备**：

```bash
# 如果有问题，回滚到上一版本
docker-compose up -d --no-deps runtime:previous
bun run migrate:down
```

## 应急预案

### 数据库宕机

1. 检查 Postgres 日志
2. 尝试重启服务
3. 从最近备份恢复
4. 切换到备用数据库

### Redis 宕机

1. 检查 Redis 日志
2. 重启 Redis 服务
3. 用户需要重新登录

### 全站宕机

1. 检查负载均衡器状态
2. 检查各服务健康状态
3. 回滚到上一个稳定版本
4. 联系运维团队
