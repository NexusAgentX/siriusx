# 仓库与信任边界

> SiriusX 按信任边界拆成可信控制面和不可信一次性沙箱运行时。

---

## 拆分原则

仓库边界必须表达安全边界，而不是只表达团队分工。长期密钥、权限判断、审计、调度状态留在可信控制面；Agent 代码生成、shell、解释器、浏览器和临时文件系统进入一次性沙箱运行时。

推荐仓库：

| 仓库 | 信任级别 | 主要职责 | 明确禁止 |
|---|---|---|---|
| `siriusx-control-plane` | 可信 | Web/API、Auth、tenant、TaskStore、Queue、lease、审计、人工审批、凭证代理、workspace revision 指针 | 执行不可信 shell、浏览器或解释器；把长期密钥挂进沙箱 |
| `siriusx-sandbox-runtime` | 不可信/一次性 | Pi AgentSession、shell、解释器、浏览器、临时文件系统、容器/MicroVM 适配、资源限制、事件上报、销毁 | 做最终权限判断；保存长期密钥；直接持有数据库或 S3 主凭证 |

---

## 跨仓库协议

控制面只向沙箱运行时发送一次性 `RunSpec`：

```typescript
interface RunSpec {
  taskId: string;
  runId: string;
  agent: { name: string; ref: string; commit: string; templateDir: string };
  baseWorkspaceRevision: string;
  workspaceManifestRef: string;
  prompt: string;
  historyWindow: Message[];
  taskSummary?: string;
  model: string;
  capabilityToken: string; // 短期、最小权限、run-scoped
  expiresAt: number;
}
```

沙箱运行时只向控制面回传可审计事件和提交建议：

```typescript
type SandboxEvent =
  | { type: "assistant_delta"; sequence: number; text: string }
  | { type: "tool_start" | "tool_update" | "tool_end"; sequence: number; payload: unknown }
  | { type: "artifact_manifest"; sequence: number; payload: ArtifactManifest }
  | { type: "workspace_commit_proposal"; sequence: number; payload: WorkspaceCommitProposal }
  | { type: "run_error" | "run_complete"; sequence: number; payload: unknown };
```

控制面负责校验事件顺序、路径 guard、artifact metadata 和 workspace commit proposal，再推进 Postgres 与 S3 状态。沙箱运行时不能直接决定 Task 完成、权限通过或 workspace revision 生效。

---

## 安全不变量

- 长期密钥只存在于控制面或云托管密钥系统，不进入沙箱镜像、环境变量或临时文件。
- 沙箱只拿 run-scoped 短期能力令牌，默认 5 分钟级别 TTL，权限限定到当前 `taskId/runId`。
- 网络、凭证、存储写入必须经过控制面的代理或预签名能力，不允许沙箱直接拼内部 object key。
- Agent session、shell、解释器、浏览器和扩展都视为不可信代码，运行结束即销毁。
- 沙箱输出是“建议”和“事件”，不是事实来源；事实来源仍是 Postgres TaskStore 和 S3 workspace revision。

---

## 故障模型

沙箱运行时崩溃时，控制面依靠 run lease 过期和 Postgres 中已持久化事件恢复；沙箱内存、临时文件和未提交输出可以丢弃。Control Worker 崩溃时，新 Control Worker 从 Postgres 和 S3 重建 `RunSpec`，重新调度到新的沙箱运行时。

---

## 落地要求

- 两个仓库共享版本化协议包或 OpenAPI/JSON Schema，至少覆盖 `RunSpec`、`SandboxEvent`、`ArtifactManifest` 和 `WorkspaceCommitProposal`。
- `siriusx-sandbox-runtime` 镜像必须独立扫描、签名和发布，安全评审不依赖控制面代码审查。
- 本地开发可以用同一个 Docker Compose 启动两个服务，但仍要保留服务边界和凭证边界。
