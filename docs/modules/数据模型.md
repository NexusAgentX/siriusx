# 数据模型

> SiriusX 平台核心数据结构定义

---

## Task（任务）

`Task` 表示长期任务容器，是用户可持续迭代的工作空间。

**🚨 P1 修正：扩展状态枚举**

```typescript
type TaskStatus = 
  | "running"           // 任务进行中
  | "running_degraded"  // 进行中但 summary 刷新连续失败（P1 新增）
  | "completed"         // 任务完成
  | "failed"            // 任务失败
  | "cancelled";        // 任务取消

interface Task {
  id: string;
  userId: string;
  tenantId: string;
  title: string;
  status: TaskStatus;   // 使用扩展后的状态
  workspaceId: string;
  workspaceRef: string;
  
  // Agent 版本锁定
  agent: string;           // Agent 名称
  agentRef: string;        // Agent 引用（如 git ref）
  agentCommit: string;     // Agent 具体 commit
  templateDir: string;     // Agent 模板目录
  
  model?: string;          // 使用的模型
  warnings?: string[];     // 可选：记录警告信息（如 summary 刷新失败）
  createdAt: number;
  completedAt?: number;
  error?: string;
}
```

### 关键特性

- **长期容器**：Task 不是一次性聊天，而是可恢复的工作空间
- **版本锁定**：创建时固化 Agent 版本，确保执行一致性
- **租户隔离**：每个 Task 关联 `userId` 和 `tenantId`
- **工作区引用**：通过 `workspaceRef` 关联持久化存储
- **状态扩展**：支持 `running_degraded` 状态（summary 刷新失败）

## TaskRun（执行单元）

`TaskRun` 表示一次用户消息触发的执行，拥有独立状态、lease、attempt、事件流和失败边界。

**🚨 P1 修正：扩展状态枚举**

```typescript
type TaskRunStatus =
  | "queued"                    // 等待 Worker 认领
  | "running"                   // Worker 正在执行
  | "completed"                 // 执行成功
  | "completed_with_warnings"   // 执行成功但有警告（P1 新增，如事件持久化部分失败）
  | "failed"                    // 执行失败
  | "cancelled";                // 执行取消

interface TaskRun {
  id: string;
  taskId: string;
  messageId: string;
  status: TaskRunStatus;        // 使用扩展后的状态
  prompt: string;
  attempt: number;
  
  // Workspace 版本控制（新增）
  base_workspace_revision: string;           // 该 run 基于的 workspace 版本
  committed_workspace_revision?: string;     // 该 run 提交的 workspace 版本
  
  // 事件序列（新增）
  last_event_sequence: number;               // 该 run 最后一个 event 的 sequence
  
  // Run lease（分布式调度）
  leaseOwnerNodeId?: string;
  leaseExpiresAt?: number;
  heartbeatAt?: number;
  
  // 时间线
  createdAt: number;
  startedAt?: number;
  completedAt?: number;
  error?: string;
  warnings?: string[];          // 可选：记录警告信息
}
```

### 关键特性

- **独立执行边界**：每个 run 有自己的状态、lease、attempt
- **乐观锁**：通过 `base_workspace_revision` 防止并发冲突
- **故障恢复**：`attempt` 字段支持重试机制
- **心跳机制**：`heartbeatAt` 用于检测 Worker 存活状态
- **状态扩展**：支持 `completed_with_warnings` 状态（事件持久化部分失败）

## TaskEvent（事件）

`TaskEvent` 必须是 run-scoped，记录执行过程中的所有事件。

```typescript
interface TaskEventEnvelope {
  id: string;
  taskId: string;
  runId: string;
  sequence: number;        // run-scoped，从 1 开始
  type: string;            // 事件类型
  payload: unknown;        // 事件内容
  createdAt: number;
}
```

### 事件约束

- `sequence` 从 1 开始，**作用域是单个 `runId`**，不是全局递增
- `(taskId, runId, sequence)` 必须唯一
- 同一 run 的持久化查询必须按 `sequence ASC` 返回
- `id` 可用于游标兼容，但断线补齐的主路径是 `afterSequence`

### 常见事件类型

**🚨 统一事件类型命名（对齐 Pi SDK 映射）**

```typescript
type TaskEventType =
  | "run_started"        // Run 开始执行
  | "assistant_delta"    // 助手回复增量
  | "thinking_delta"     // 思考过程增量
  | "tool_start"         // 工具调用开始
  | "tool_update"        // 工具调用进度更新
  | "tool_end"           // 工具调用结束
  | "turn_complete"      // 本轮完成
  | "error"              // 执行错误（统一使用 error，不是 task_error）
  | "run_cancelled";     // Run 被取消
```

**关键说明**：
- 统一使用 `error`（不是 `task_error`）与 API 流式事件保持一致
- 包含完整事件类型列表（包括 `thinking_delta` 和 `tool_update`）
- 所有事件类型都需要在持久化层支持

## TaskExecutionLock（执行锁）

保证同一个 Task 默认串行执行。

```typescript
interface TaskExecutionLock {
  taskId: string;
  runId: string;
  ownerNodeId: string;
  expiresAt: number;
  heartbeatAt: number;
}
```

### 用途

- 防止同一个 Task 的多个 run 并发执行
- Control Worker 崩溃后，锁会自动过期
- 其他 Control Worker 可以接管过期的 run

## SandboxLease（沙箱租约）

**🚨 阶段 0-2 不实现此模型**

在阶段 0-2 的 Stateless Sandbox 策略下，**不实现 `SandboxLease` 表、model、interface 或 repository**。

代码库中不应包含此接口或相关实现。

**未来扩展**（阶段 4+）：如果引入 task-scoped sandbox pool，才需要此表。

## TaskMessage（消息）

```typescript
interface TaskMessage {
  id: string;
  taskId: string;
  runId?: string;          // 关联的 run（如果是系统消息可能为空）
  role: "user" | "assistant" | "tool";  // 🚨 对齐 schema
  parts: MessagePart[];    // 🚨 对齐 schema：parts 数组而非 content 字符串
  createdAt: number;
}

type MessagePart = 
  | { type: "text"; text: string }
  | { type: "tool-call"; toolCallId: string; name: string; input: unknown }
  | { type: "tool-result"; toolCallId: string; result: unknown };
```

## TaskArtifact（产物）

```typescript
interface TaskArtifact {
  taskId: string;
  path: string;            // 文件路径
  kind: "document" | "file" | "code" | "image";  // 🚨 对齐 schema
  status: "generated" | "modified" | "template" | "deleted";  // 🚨 对齐 schema
  workspace_revision: string;
  object_key: string;      // S3 对象键
  etag: string;            // S3 ETag，用于校验
  content_type: string;
  size_bytes: number;
  updated_at: number;
}
```

### 关键特性

- **版本追踪**：通过 `workspace_revision` 关联特定版本
- **S3 存储**：大文件内容存储在 S3，Postgres 只保存元数据
- **校验机制**：通过 `etag` 确保数据完整性

## TaskSummary（任务摘要）

```typescript
interface TaskSummary {
  taskId: string;
  summary: string;         // 上下文摘要
  runCount: number;        // 已完成的 run 数量
  tokenCount: number;      // 累计 token 数
  artifactCount: number;   // 产物数量
  updatedAtRunCount: number;  // 上次更新摘要时的 run 数量
  consecutiveFailures: number; // 连续失败次数
  lastSuccessfulRunId: string | null; // 最后一次成功刷新的 run（对齐 STAGE0_SPEC §10.4）
  createdAt: number;
  updatedAt: number;
}
```

### 用途

- 压缩上下文，避免每次都读取完整历史
- 刷新策略（对齐 STAGE0_SPEC §10.3）：
  - 每个 run 完成后触发一次**异步** refresh（system background job，**不阻塞** run / message）
  - 参考阈值：10 runs / 30k tokens / 50 files
- 失败处理：递增 `consecutiveFailures` 并写 warning；**不自动切换** task 到 `running_degraded`（§5.7）

## WorkspaceRevision（工作区版本）

```typescript
interface WorkspaceRevision {
  id: string;
  task_id: string;
  workspace_ref: string;
  revision: string;        // 版本号（单调递增或内容寻址）
  base_revision?: string;  // 基于的版本
  
  // 状态机（两阶段提交）
  status: "pending" | "uploading" | "committed" | "failed";
  
  // S3 引用
  manifest_object_key: string;
  manifest_etag: string;
  
  committed_run_id?: string;
  created_at: number;
  committed_at?: number;   // committed 状态时间
  expires_at?: number;     // pending/uploading 状态的 TTL
}
```

### 两阶段提交

1. **PG 先行**：创建 `pending` 状态的 revision
2. **上传 S3**：状态转为 `uploading`
3. **验证可见性**：S3 read-after-write check
4. **提交成功**：状态转为 `committed`，推进 Task 的 `latest_workspace_revision`

## 状态机总结

### Task 状态转换

```
running → running_degraded (summary 失败 3 次)
running → completed (用户关闭)
running → failed (致命错误)
running → cancelled (用户取消)
```

### TaskRun 状态转换

```
queued → running → completed
                → completed_with_warnings (事件持久化部分失败)
                → failed
                → cancelled
```
