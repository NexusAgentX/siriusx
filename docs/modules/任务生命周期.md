# Task 生命周期

## Task 创建

`POST /api/task` 只创建 Task 和 workspace，不消费用户 prompt，不创建 run，不创建 Pi session，不创建 sandbox。

### 请求体

```json
{
  "title": "支付系统架构设计",
  "agent": "architecture",
  "model": "deepseek-v4-flash"
}
```

### 后端处理流程

```
derive userId and tenantId from AuthContext
  -> resolve AgentCatalog entry
  -> lock agent metadata into Task: agent, agentRef, agentCommit, templateDir
  -> create durable workspaceRef
  -> copy agent template into workspace
  -> create Task(status=running)
  -> return taskId
```

### 说明

- 任意 API 节点都可以处理 Task 创建。
- Prompt 必须通过 `/message` 创建 run。
- 前端可以在 Task 创建成功后立即调用 `/message` 发送首条 prompt，但不能把首条 prompt 隐式塞回 `POST /api/task`。
- `userId`、`tenantId`、`agentRef`、`agentCommit`、`templateDir` 都由后端确定，body 中出现同名字段必须忽略或拒绝。

## 发送消息和创建 Run

`POST /api/task/:taskId/message` 创建一轮执行。

### 后端处理流程

```
validate task is open
  -> append user message
  -> create TaskRun(status=queued, attempt=1)
  -> enqueue TaskTurnJob(taskId, runId, messageId)  // prompt 从 message 读取，不入队（STAGE0_SPEC §6.1）
  -> subscribe ResultBus channel for this run
  -> stream run-scoped events

### ResultBus 通道

ResultBus channel 必须 run-scoped：

```
task:{taskId}:run:{runId}
```

这样同一个 task 的快速连续消息不会互相串流。

### 连续消息处理策略

**🚨 阶段 0-2 串行策略**：同一 Task 同时只允许一个 active run。

- 如果新消息到达时已有 active run（queued 或 running），`POST /message` **必须返回 `409 RUN_CONFLICT`**。
- 响应 body 包含当前 active `runId` 和状态，前端显示"任务执行中，请稍后重试"。
- 前端必须等待当前 run 完成（`turn_complete` 或 `error`），才能发送下一条消息。
- **不实现 run 队列**：简化阶段 0-2 实现，避免队列管理复杂度。
- `/message` stream 只能发送当前新建 run 的事件；不能混入同一 task 其他 run 的 delta。

**未来扩展（阶段 3+）**：
- 实现 per-task run 队列
- `POST /message` 创建 run 并入队，返回 `run_queued` 事件
- Worker 串行消费同一 Task 的 run 队列
