# 工作区提供器（WorkspaceProvider）

**🚨 阶段 0 必须实现 S3 Provider，否则多节点无法工作**

多节点 SaaS 不能依赖某台 Worker 的本地磁盘作为唯一 workspace。生产目标是 S3-backed workspace：Worker 可以使用本地目录作为 materialized cache，但 durable source of truth 是 S3 revision manifest 和 blobs，最新 revision pointer 存在 Postgres。

**阶段划分：**
- **阶段 0**：实现基础 S3WorkspaceProvider（简化版，全量上传/下载）
- **阶段 1-2**：基于 S3 Provider 安全扩展到 2+ Worker 节点
- **阶段 3**：S3 性能优化（增量上传、本地缓存、并行传输）

**本地 workspace 仅用于单节点开发环境**，不支持多 Worker 部署。

## workspaceRef

`workspaceRef` 是 durable workspace 的稳定引用，不是某台机器上的本地路径。

```
workspaceRef = workspace://<tenantId>/<taskId>/<revision>
```

约束：

- `<tenantId>` 和 `<taskId>` 用于权限和隔离。
- `<revision>` 表示 durable checkpoint 版本；如果 backend 不支持版本，也必须能映射到最新 durable 状态。
- 本地 materialize 后的路径只在当前 Worker 生命周期内有效，不能写入 TaskStore 当作事实来源。
- `workspaceRef` 可以映射到 PG `tasks.latest_workspace_revision` 和 S3 manifest；API 响应不暴露本地路径或裸 S3 key。

## Provider 接口

```typescript
interface WorkspaceProvider {
  create(input: {
    tenantId: string;
    taskId: string;
    templateDir: string;
  }): Promise<{ workspaceRef: string }>;

  materialize(input: {
    workspaceRef: string;
    taskId: string;
    runId: string;
  }): Promise<{ workspaceDir: string; baseRevision: string }>;

  commit(input: {
    workspaceRef: string;
    workspaceDir: string;
    taskId: string;
    runId: string;
    baseRevision: string;
  }): Promise<{ workspaceRef: string; revision: string }>;
}
```

## S3 Manifest 形态

```json
{
  "taskId": "task-123",
  "revision": "00000003",
  "baseRevision": "00000002",
  "committedRunId": "run-003",
  "createdAt": 1718100000000,
  "files": [
    {
      "path": "docs/ARCHITECTURE.md",
      "objectKey": "tenants/t-1/tasks/task-123/workspace/blobs/sha256:abc...",
      "sha256": "abc...",
      "sizeBytes": 18420,
      "contentType": "text/markdown"
    }
  ]
}
```

## Durability 要求

- Control Worker 在标记 run `completed` 之前，必须先完成 workspace `commit()` 并拿到 storage backend 的成功确认。
- 下一个 run materialize 同一 Task 时，必须能看到上一个 completed run 的 workspace 改动；这是同 Task 串行执行的 read-after-write 保证。
- 对象存储或远程同步可以是 eventually consistent，但 `commit()` 返回前必须通过 S3 manifest + PG version pointer 保证后续 materialize 读到同一 revision。
- Control Worker 崩溃时，未 commit 的 local cache 可以丢弃；已 commit 的 revision 必须可被其他 Control Worker materialize。
- Artifact indexing 只能基于已 commit 或即将随 run completion 一起提交的 workspace revision。
- `commit()` 必须先上传 S3 blobs 和 manifest，再推进 PG latest pointer；不能先写 PG 再补传 S3。
- Manifest 中只允许 workspace 相对路径，必须拒绝绝对路径、`../`、symlink escape 和隐藏系统目录。

## 后端实现选项

- S3 / 对象存储 后端 workspace
- 本地缓存 + S3 持久源同步
- Git 后端 workspace 快照
- 共享网络卷（开发或过渡期可用）

## 实现边界

- 任何 Control Worker 都能从 `workspaceRef` materialize workspace。
- local cache 可以存在，但必须可丢弃。
- 本地文件路径不能出现在 API 响应中。
- workspace provider 必须做 path guard，防止 `../`、symlink escape 和隐藏目录越权读取。
- `listFiles()` 和 artifact 读取走 PG artifact index 或 S3 manifest，不通过递归扫描 Worker 本地 cache 作为事实来源。
