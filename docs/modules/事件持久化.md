# 事件持久化优化

> 实时流式传输 + 批量持久化 + 降级处理

---

## 概述

实时流可以保持细粒度，但落库不能每个 `text_delta` 一次 insert。

**核心策略：内存缓冲 + 异步批量持久化**

**🚨 阶段 0-2 限制**：
- 内存缓冲仅用于同一 Control Worker 进程内防丢失
- Control Worker 崩溃后，未持久化事件**已丢失**
- 新 Control Worker 只能从 Postgres 恢复已持久化的事件
- **不能保证不丢失**，只能尽力而为

---

## 架构流程

```
Pi Event → EventSequencer (分配 sequence)
         → Memory Buffer (Control Worker 进程内缓存)
         → ResultBus (实时发布)
         → EventPersistenceBuffer (批量缓冲)
         → TaskStore (异步批量写入 PG)
            ↓ flush 失败（阶段 0-2：仅 best-effort 重试 + completed_with_warnings）
         → (阶段 3+ 才引入 S3 Backup)
```

---

## 详细流程

### 1. Sequence 分配（原子递增）

从 PG 的 `task_runs.last_event_sequence` 字段原子递增：

```sql
UPDATE task_runs 
SET last_event_sequence = last_event_sequence + 1 
WHERE id = $1 
RETURNING last_event_sequence;
```

🚨 **阶段 0 必须每次原子递增，禁止批量预留**（见 STAGE0_SPEC §7.4.1）。批量预留会扩大 sequence gap，且无法保证 run 跨 Worker 时的递增语义。

---

### 2. 内存缓冲（防止进程内丢失）

**🚨 P0 修正：明确为进程内缓存，不支持跨节点恢复**

Control Worker 进程维护内存中的事件缓冲区：

```typescript
class EventPersistenceService {
  // 仅用于同一 Control Worker 进程内的短期缓存
  // Control Worker 崩溃后，新 Control Worker 无法访问此缓存
  private memoryBuffer: Map<string, Event[]> = new Map();
  
  async appendEvent(runId: string, event: Event) {
    // 1. 写入内存缓冲（best-effort，同进程内防丢失）
    const buffer = this.memoryBuffer.get(runId) || [];
    buffer.push(event);
    this.memoryBuffer.set(runId, buffer);
    
    try {
      // 2. 批量写入 Postgres（事实来源）
      await this.flushToPG(runId);
      
      // 3. 成功后清理内存缓冲
      this.memoryBuffer.delete(runId);
    } catch (error) {
      // 4. 持久化失败：保留内存缓冲，等待重试或降级
      logger.error(`Event persistence failed for run ${runId}`, error);
    }
  }
}
```

**关键限制**：
- 内存缓冲仅用于**同一 Control Worker 进程内**的短期缓存
- Control Worker 崩溃后，新 Control Worker **只能从 Postgres** 恢复已持久化的事件
- 内存缓冲未持久化的事件会丢失（阶段 0-2 **不实现** S3 backup；阶段 3+ 才引入）

---

### 3. 实时流发布（用户可见）

立即发布到 ResultBus，不等待 PG 写入：

```typescript
await resultBus.publish(`task:${taskId}:run:${runId}`, {
  sequence,
  type: event.type,
  payload: event.payload,
});
```

---

### 4. 批量缓冲（合并高频 delta）

**🚨 P1 修正：明确 Durable Sequence 分配规则**

`EventPersistenceBuffer` 持有同一批 sequenced events，可以在落库前合并高频 delta。

**🚨 阶段 0-2 简化策略**：
- Realtime 和 Durable 事件使用**相同类型**
- 直接使用 Pi SDK 映射的类型：`assistant_delta`, `tool_start`, `tool_end`
- 不实现 `assistant_delta_batch` / `thinking_delta_batch`
- 高频 delta 按原样持久化，前端自行合并

```typescript
// 实时事件和持久化事件统一类型
{
  type: "assistant_delta",  // Pi SDK 映射
  delta: "hello",
  sequence: 42  // 分配后直接持久化
}

// 查询示例：afterSequence=15 → 返回 sequence > 15 的所有事件
```

**未来优化**（阶段 3+）：
- 实现批次合并：`assistant_delta_batch`
- 分离 realtime 和 durable sequence

**触发 flush**：

```
flush interval: 100ms
max batch size: 50 events
max text batch: 1KB
```

---

### 5. 批量写入 PG（异步持久化）

使用 `INSERT ... ON CONFLICT DO NOTHING` 保证幂等：

```sql
-- 🚨 对齐 schema：列名是 payload，不是 payload_jsonb
INSERT INTO task_events (task_id, run_id, sequence, type, payload, created_at)
VALUES ($1, $2, $3, $4, $5, EXTRACT(EPOCH FROM NOW())::bigint * 1000)
ON CONFLICT (task_id, run_id, sequence) DO NOTHING;
```

批量写入成功后，从内存缓冲中删除对应 events。

---

### 6. 失败降级处理（尽力而为）

**🚨 阶段 0-2 简化策略**

如果 PG 写入失败，重试 3 次（统一退避：5s, 30s, 120s；与 STAGE0_SPEC §6.4 / Worker执行.md 一致）。

重试仍失败后，标记 run 为 `completed_with_warnings`：

```typescript
await taskStore.updateRun(runId, {
  status: "completed_with_warnings",
  warnings: ["event_persistence_incomplete"],
});
```

**关键限制**：
- 阶段 0-2 **不实现** S3 backup 恢复
- `events_backup_s3_key` / `incomplete_sequence_ranges` 未在 schema 中定义
- Control Worker 崩溃后，内存缓冲中未持久化的事件**已丢失**
- 新 Control Worker 只能从 Postgres 恢复已持久化的事件
- **不能保证不丢失**，只能尽力而为

**未来扩展**（阶段 3+）：
- 扩展 schema 添加 `events_backup_s3_key` 列
- 实现 S3 backup 上传和恢复
- 前端补齐缺失事件逻辑

## Control Worker 崩溃恢复

**🚨 P0 修正：新 Control Worker 只能从 Postgres 恢复**

```typescript
async function recoverRun(run: TaskRun): Promise<void> {
  // 1. 检查 attempt 上限（阶段 0 不投递 DLQ，超过上限直接 failed）
  if (run.attempt >= MAX_ATTEMPTS) {
    await taskStore.updateRun(run.id, {
      status: 'failed',
      error: 'Max attempts exceeded',
    });
    return;
  }

  // 2. 递增 attempt，重新入队；新 Control Worker claim 后从 Postgres 重建上下文
  //   （崩溃恢复路径由 LeaseSweeper + Queue 驱动，见 STAGE0_SPEC §6.1.1 / 队列与租约.md）
  //   本函数不直接创建 Pi session，也不直接执行 run。
  run.attempt++;
  await taskStore.updateRun(run.id, { attempt: run.attempt });
  await queue.enqueue({
    runId: run.id,
    attempt: run.attempt,
  }, {
    delay: RETRY_DELAYS_MS[run.attempt - 2] ?? 0, // 统一退避：5s, 30s, 120s
  });

  // 3. 新 Control Worker claim 后会：
  //   - 从 Postgres 读取 run 的事件、消息、task summary、latest workspace revision
  //   - 构造新的 RunSpec，交给沙箱运行时执行
  //   - 使用 ON CONFLICT DO NOTHING 避免重复写入已持久化事件
}
```

## 并发和顺序安全

- 同一 run 的 sequence 由 active run owner 的 `EventSequencer` 单调分配（基于 PG 原子递增）
- `task_runs` 表增加 `last_event_sequence` 字段，每次分配后推进
- `task_events` 建唯一约束 `(task_id, run_id, sequence)`
- 批量写入必须保持输入顺序
- **🚨 阶段 0-2**：每个事件直接分配 sequence 并持久化，realtime 和 durable 使用相同 sequence
- 多个 Control Worker 可以同时写不同 run 的事件；不要求全局事件顺序，只要求 run-scoped 顺序
- Control Worker 崩溃后，新 Control Worker 从 Postgres 恢复已持久化的事件，使用 `ON CONFLICT DO NOTHING` 避免重复写入

## TaskStore 接口

**🚨 完整契约定义**：TaskStore 是核心存储接口，被 Worker、API、队列管理器依赖

```typescript
interface TaskStore {
  // ============================================================================
  // Task 管理
  // ============================================================================
  
  createTask(input: {
    id: string;
    userId: string;
    tenantId: string;
    title: string;
    agent: string;
    agentRef?: string;
    agentCommit?: string;
    templateDir?: string;
    workspaceId: string;
    workspaceRef?: string;
  }): Promise<Task>;
  
  getTask(taskId: string): Promise<Task | null>;
  
  updateTask(taskId: string, updates: Partial<Task>): Promise<void>;
  
  updateTaskWorkspaceRevision(input: {
    taskId: string;
    newRevision: string;
    expectedCurrentRevision: string;  // 乐观锁
  }): Promise<boolean>;  // false = 冲突
  
  // ============================================================================
  // Run 管理
  // ============================================================================
  
  createRun(input: {
    id: string;
    taskId: string;
    messageId?: string;
    prompt: string;
    baseWorkspaceRevision?: string;
  }): Promise<TaskRun>;
  
  getRun(runId: string): Promise<TaskRun | null>;
  
  updateRun(runId: string, updates: Partial<TaskRun>): Promise<void>;
  
  // ============================================================================
  // Run Lease 管理（队列与租约依赖）
  // ============================================================================
  
  claimRun(input: {
    runId: string;
    workerNodeId: string;
    leaseExpiresAt: number;
  }): Promise<boolean>;  // false = 已被其他 Control Worker claim
  
  extendRunLease(input: {
    runId: string;
    workerNodeId: string;
    newExpiresAt: number;
  }): Promise<boolean>;  // false = lease 已被接管或 run 已完成
  
  releaseRunLease(input: {
    runId: string;
    workerNodeId: string;
  }): Promise<void>;
  
  // ============================================================================
  // 事件持久化
  // ============================================================================
  
  appendEvents(input: {
    taskId: string;
    runId: string;
    events: Array<{
      sequence: number;       // Durable sequence（落库）
      type: string;
      payload: unknown;
    }>;
  }): Promise<void>;
  
  // 查询 run 的事件（支持断点续传）
  getRunEvents(input: {
    taskId: string;
    runId: string;
    afterSequence?: number;  // Durable sequence
    limit?: number;
  }): Promise<TaskEventEnvelope[]>;
  
  // ============================================================================
  // 消息管理
  // ============================================================================
  
  createMessage(input: {
    id: string;
    taskId: string;
    runId?: string;
    role: 'user' | 'assistant' | 'tool';
    parts: unknown[];
  }): Promise<void>;
  
  getTaskMessages(taskId: string, limit?: number): Promise<TaskMessage[]>;
  
  // ============================================================================
  // Artifact 管理（Worker 执行依赖）
  // ============================================================================
  
  upsertArtifact(input: {
    taskId: string;
    path: string;
    kind: 'document' | 'file' | 'code' | 'image';
    status: 'generated' | 'modified' | 'template' | 'deleted';
    workspaceRevision?: string;
    objectKey?: string;
    etag?: string;
    contentType?: string;
    sizeBytes?: number;
  }): Promise<void>;
  
  getTaskArtifacts(taskId: string): Promise<TaskArtifact[]>;
}
```

**关键说明**：
- 所有写操作必须幂等（支持重试）
- Lease 相关方法返回 `boolean` 表示是否成功（用于乐观锁检测）
- `updateTaskWorkspaceRevision` 使用乐观锁防止并发冲突

## 持久化事件类型

**🚨 阶段 0-2**：直接使用 Pi SDK 映射类型，不实现批次合并

完整事件类型列表：
- `run_started` — Run 开始执行
- `assistant_delta` — 助手回复增量（高频，原样持久化）
- `thinking_delta` — 思考过程增量
- `tool_start` — 工具调用开始
- `tool_update` — 工具调用进度更新
- `tool_end` — 工具调用结束
- `turn_complete` — 本轮完成
- `error` — 执行错误（统一使用 error，不是 task_error）
- `run_cancelled` — Run 被取消

**未来扩展**（阶段 3+）：
- `assistant_delta_batch` — 合并多个 text_delta
- `thinking_delta_batch` — 合并多个 thinking_delta
- 节流工具进度更新（减少 `tool_update` 频率）
- `run_retrying` — Run 重试事件

---

## 关键改进

- **内存缓冲是 best-effort**：仅用于同一 Control Worker 进程内防丢失，不支持跨节点恢复
- **Postgres 是事实来源**：Control Worker 崩溃后，新 Control Worker 只从 Postgres 恢复
- **降级而非失败**：持久化失败时标记为 `completed_with_warnings`，而不是让整个 run 失败
- **🚨 阶段 0-2 不实现 S3 backup**：未来扩展（阶段 3+）才实现 S3 事件备份和前端补齐
- **阶段 0-2 sequence 分配**：每个事件直接分配 sequence 并持久化，不实现批次合并
- **幂等性保证**：`ON CONFLICT DO NOTHING` + `(task_id, run_id, sequence)` 唯一约束防止重复写入
