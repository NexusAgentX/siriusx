# 分布式队列和 Run Lease

## 分布式队列设计

生产队列必须支持以下能力：

- **claim with visibility timeout** — Control Worker 领取 job 后，队列将其隐藏，防止被其他 Control Worker 重复消费
- **heartbeat / lease extension** — Control Worker 在消费期间定期发送心跳，动态延长 lease 以避免 job 被重新投递
- **retry with attempt count** — 每次重试记录 attempt 次数，支持指数退避
- **dead-letter queue** — 超过最大重试次数后进入死信队列
- **delayed retry** — 失败后延迟一定时间再投递，结合退避策略

**🚨 阶段 0-2 不实现 per-task run 队列**：
- 同一 Task 已有 active run 时，`POST /message` 返回 `409 RUN_CONFLICT`
- 前端必须等待当前 run 完成才能发送下一条消息
- 简化实现，避免 run 队列管理复杂度
- 未来扩展（阶段 3+）才实现 per-task run 队列

## Run Lease 心跳机制

Run Lease 管理是 Control Worker 与队列之间保持会话的机制。Control Worker claim 一个 Run 后获得一段时间的排他 lease，并通过周期性心跳延长 lease。如果 Control Worker 崩溃或卡死，心跳停止，lease 自动过期，Run 可被其他 Control Worker 接管。

```typescript
class RunLeaseManager {
  private heartbeatIntervalMs = 30_000; // 30s 心跳间隔
  private leaseTimeoutMs = 90_000; // 90s lease 超时（3 倍心跳间隔）
  private heartbeatTimers = new Map<string, NodeJS.Timeout>();
  
  async claimRun(runId: string, workerNodeId: string): Promise<boolean> {
    // 1. 尝试 claim run（乐观锁）
    const claimed = await taskStore.claimRun({
      runId,
      workerNodeId,
      leaseExpiresAt: Date.now() + this.leaseTimeoutMs,
    });
    
    if (!claimed) {
      return false; // 被其他 Control Worker claim 了
    }
    
    // 2. 启动心跳线程
    this.startHeartbeat(runId, workerNodeId);
    
    return true;
  }
  
  private startHeartbeat(runId: string, workerNodeId: string) {
    const timer = setInterval(async () => {
      try {
        // 发送心跳，延长 lease
        const extended = await taskStore.extendRunLease({
          runId,
          workerNodeId,
          newExpiresAt: Date.now() + this.leaseTimeoutMs,
        });
        
        if (!extended) {
          // Lease 已被其他 Control Worker 接管或 run 已完成
          this.stopHeartbeat(runId);
          throw new LeaseRevokedError(`Run ${runId} lease was revoked`);
        }
        
        logger.debug(`Heartbeat sent for run ${runId}`);
        
      } catch (error) {
        logger.error(`Heartbeat failed for run ${runId}`, error);
        this.stopHeartbeat(runId);
        throw error;
      }
    }, this.heartbeatIntervalMs);
    
    this.heartbeatTimers.set(runId, timer);
  }
  
  stopHeartbeat(runId: string) {
    const timer = this.heartbeatTimers.get(runId);
    if (timer) {
      clearInterval(timer);
      this.heartbeatTimers.delete(runId);
    }
  }
  
  async releaseRun(runId: string, workerNodeId: string) {
    // 停止心跳
    this.stopHeartbeat(runId);
    
    // 释放 lease
    await taskStore.releaseRunLease({
      runId,
      workerNodeId,
    });
  }
}
```

## SQL 实现（Postgres）

```sql
-- Claim run（乐观锁）
UPDATE task_runs
SET 
  lease_owner_node_id = $1,
  lease_expires_at = EXTRACT(EPOCH FROM NOW())::bigint * 1000 + 90000,  -- 当前时间 + 90s
  heartbeat_at = EXTRACT(EPOCH FROM NOW())::bigint * 1000,
  status = 'running'
WHERE 
  id = $2
  AND status = 'queued'
  AND (lease_expires_at IS NULL OR lease_expires_at < EXTRACT(EPOCH FROM NOW())::bigint * 1000)
RETURNING *;

-- Extend lease（心跳）
UPDATE task_runs
SET 
  lease_expires_at = $1,  -- 传入计算好的 Unix ms
  heartbeat_at = EXTRACT(EPOCH FROM NOW())::bigint * 1000
WHERE 
  id = $2 
  AND lease_owner_node_id = $3
  AND status = 'running'
RETURNING *;

-- Release lease
UPDATE task_runs
SET 
  lease_owner_node_id = NULL,
  lease_expires_at = NULL
WHERE 
  id = $1 
  AND lease_owner_node_id = $2;
```

**关键 SQL 设计要点**：

- **乐观锁（Conditional Update）** — 所有更新操作都通过 `WHERE` 条件校验当前状态，避免并发冲突
- **Claim 条件** — 要求 `status = 'queued'` 且 lease 已过期或为空，确保同一时间只有一个 Worker 持有 lease
- **Extend 条件** — 要求 `lease_owner_node_id = $3` 且 `status = 'running'`，防止非 owner 节点继约
- **Release 条件** — 要求 `lease_owner_node_id = $2`，避免误释放其他 Control Worker 的 lease

## LeaseSweeper（过期 lease 检测与重新入队）

LeaseSweeper 是独立运行的守护服务，定期扫描过期 lease，将超时的 Run 重新放回队列。这是整个系统的 **Dead Man's Switch** 机制——即使 Worker 进程存活但卡死，超时后也会自动释放 lease。

```typescript
class LeaseSweeper {
  async sweep() {
    // 1. 查找过期的 run lease
    const expiredRuns = await taskStore.findExpiredRunLeases();
    
    for (const run of expiredRuns) {
      logger.warn(`Run ${run.id} lease expired, reclaiming...`);
      
      // 2. 将 run 重新放回队列
      await taskQueue.enqueue({
        taskId: run.taskId,
        runId: run.id,
        messageId: run.messageId,
        attempt: run.attempt + 1, // 递增 attempt
      });
      
      // 3. 更新 run 状态为 queued
      await taskStore.updateRun(run.id, {
        status: "queued",
        leaseOwnerNodeId: null,
        leaseExpiresAt: null,
      });
    }
  }
}

// 每分钟运行一次
setInterval(() => leaseSweeper.sweep(), 60_000);
```

**SQL 查询过期 lease**：

```sql
SELECT id, task_id, lease_owner_node_id, attempt
FROM task_runs
WHERE 
  status = 'running'
  AND lease_expires_at < EXTRACT(EPOCH FROM NOW())::bigint * 1000;
-- 🚨 lease_expires_at 已融合心跳（每次 heartbeat 都推进 lease_expires_at = now + 90s）。
-- 只要 lease 过期即可重新入队，无需额外 heartbeat_at 阈值（原 2 分钟判断会延迟 30s+ 回收，与 90s lease 语义冲突）。

## Worker 执行流程集成心跳

```typescript
class TaskWorker {
  async executeRun(job: TaskTurnJob) {
    const { taskId, runId, messageId, prompt } = job;
    
    // 1. Claim run lease（启动心跳）
    const claimed = await this.leaseManager.claimRun(runId, this.nodeId);
    if (!claimed) {
      throw new Error(`Failed to claim run ${runId}`);
    }
    
    try {
      // 2. Acquire task execution lock（防止同 task 并发）
      const lock = await this.acquireTaskLock(taskId, runId);
      
      try {
        // 3. Record base workspace revision
        const task = await taskStore.getTask(taskId);
        await taskStore.updateRun(runId, {
          base_workspace_revision: task.latest_workspace_revision,
        });
        
        // 4. Materialize workspace
        const workspaceDir = await this.materializeWorkspace(taskId, task.latest_workspace_revision);
        
        // 5. Create Pi session
        const piSession = await this.createPiSession(taskId, workspaceDir);
        
        // 6. Execute run（心跳在后台持续发送）
        for await (const event of piSession.run(prompt)) {
          await this.eventPersistence.handle(event, runId);
        }
        
        // 7. Commit workspace
        await this.commitWorkspace(taskId, runId, workspaceDir);
        
        // 8. Mark run completed
        await taskStore.updateRun(runId, { status: "completed" });
        
      } finally {
        // 释放 task lock
        await this.releaseTaskLock(taskId, runId);
      }
      
    } catch (error) {
      // 标记 run failed
      await taskStore.updateRun(runId, {
        status: "failed",
        error: error.message,
      });
      
    } finally {
      // 停止心跳并释放 lease
      await this.leaseManager.releaseRun(runId, this.nodeId);
    }
  }
}
```

## 关键设计决策

- **心跳间隔 30s，lease 超时 90s** — 给 Worker 3 次心跳机会，应对短暂网络抖动或调度延迟
- **独立心跳线程** — 不依赖事件流，即使 Pi SDK hang 也能持续发送心跳
- **Lease Sweeper** — 独立服务每分钟检测过期 lease，自动重新入队，不依赖 Worker 进程
- **乐观锁防止冲突** — 使用 `WHERE lease_owner_node_id = ? AND status = 'running'` 条件更新，天然防止并发冲突
- **Dead Man's Switch** — 即使 Worker 进程存活但卡死，超时后自动释放 lease

## Run Claim 流程

```
claim job from queue
  -> update TaskRun leaseOwnerNodeId, leaseExpiresAt, heartbeatAt
  -> start heartbeat thread (30s interval)
  -> acquire TaskExecutionLock(taskId)
  -> execute run
  -> stop heartbeat and release lock
```

## Control Worker 崩溃恢复流程

```
heartbeat stops
  -> run lease expires (90s timeout)
  -> LeaseSweeper detects expired lease (next sweep cycle)
  -> queue makes job claimable again
  -> another Worker increments attempt
  -> reconstruct context from TaskStore + workspace
```

## Run 重试策略

- **最大 attempt** — 默认 3 次，首次执行记为 attempt 1，后续重试递增
- **重试间隔** — 指数退避加抖动：5s、30s、2m，上限由队列 adapter 配置
- **可重试错误**：
  - Worker crash
  - Run lease timeout
  - Queue visibility timeout
  - 临时网络错误
  - Workspace materialize 临时失败
  - Sandbox create/exec timeout
  - ResultBus 临时不可用
- **不可重试错误**：
  - Pi SDK auth failure
  - Model/provider 配置错误
  - Agent 模板缺失
  - 用户权限不足
  - Task 已关闭
  - Prompt 校验失败
  - 确定性的 path guard violation
  - **WorkspaceConflictError**（workspace revision 冲突）
- **重试行为** — 重试前必须重新 materialize workspace，并重新创建本次 run 的短生命周期 Pi session；默认不在重试时自动降级 model 或切换 sandbox provider（这会改变结果可解释性），如需降级必须记录为显式 policy decision
- **超过最大 attempt** — Run 标记为 `failed`，Task 仍保持 open，用户可以修改 prompt 后创建新 Run
