# 短生命周期会话设计（Short-Lived Sessions）

> 为什么 SiriusX 不让 AI "一直守着你"，而是"做完就走，需要时再回来翻记录"

---

## 🤔 "不接着记忆"是什么意思？

### 两种方式对比

#### ❌ 方式 A：接着记忆（传统聊天机器人）

想象你在和 ChatGPT 聊天：

```
你：帮我写一个购物清单
AI：好的，牛奶、面包、鸡蛋...
你：再加点水果
AI：好的，我记得你要牛奶、面包、鸡蛋，现在加上苹果、香蕉
```

AI 把整个对话历史都保存在"脑子里"（内存中），**一直在线等着你**继续问。

**问题**：如果 100 个人同时聊天，AI 要同时记住 100 段对话，每段几千字，服务器会爆掉💥

---

#### ✅ 方式 B：不接着记忆（SiriusX 的方式）

想象你去看医生：

**第一次看诊（今天上午）**：
- 你：我头疼
- 医生：开了药 A，写在病历本上 📖

**第二次看诊（今天下午）**：
- 你：还是头疼，能换个药吗？
- 医生：先翻开你的病历本 → 看到上午开了药 A → 理解你的情况 → 开新药 B

医生**不是一直坐在你家门口等你**（那样太浪费了），而是看完病就去看下一个患者。你再来的时候，翻病历恢复记忆。

## SiriusX 是怎么做的？

### 第一轮（run-001）

```
小明问问题
  ↓
系统分配一个 Worker（工人）
  ↓
Worker 启动一个 AI 会话（医生坐下看诊）
  ↓
AI 生成文档
  ↓
保存到数据库 📁
  ↓
Worker 关闭 AI 会话 ❌（医生去看下一个患者了）
```

### 第二轮（run-002）

```
小明继续问问题
  ↓
系统分配一个 Worker（可能是同一个，也可能是另一个）
  ↓
Worker 先做这些事：
  1. 从数据库读取 📖 第一轮的对话和文档
  2. 启动一个**全新的** AI 会话
  3. 把历史"喂"给 AI
  ↓
AI 理解上下文后，修改文档
  ↓
保存到数据库 📁
  ↓
Worker 再次关闭 AI 会话 ❌
```

## 为什么要这样设计？

### 资源节省

| 方式 | 100 个用户同时使用 | 说明 |
|---|---|---|
| 接着记忆 | 100 个 AI 一直在线，占 200GB 内存 | 服务器爆炸💥 |
| 不接着记忆 | 同时最多 10 个在线，占 20GB 内存 | 轻松应对 ✅ |

### 可扩展性

- ✅ 可以随时增加 Control Worker 数量
- ✅ 任何 Control Worker 都能处理任何任务（只要读数据库）
- ✅ 某个 Control Worker 崩溃了，另一个 Control Worker 接手（就像医生生病了，换另一个医生看你的病历）

## "病历本"里有什么？

SiriusX 的"病历本"（数据库）保存：

### 1. 对话历史

```
[第1轮] 小明：为支付系统设计架构
[第1轮] AI：我建议使用微服务架构...
[第2轮] 小明：改成 REST 同步调用
```

### 2. 生成的文档

```
📄 architecture-plan.md（版本 1）
📄 architecture-plan.md（版本 2，已修改）
```

### 3. 任务状态

```
Task ID: task-123
状态: 进行中
已完成轮次: 2
```

### 4. 上下文摘要（每 10 轮自动生成）

```
小明在设计支付系统，最初使用事件驱动架构，
后改为 REST 同步调用，目前考虑加入 Redis 缓存层。
```

## 常见疑问

### Q: AI 会"忘记"之前的对话吗？

**不会！**

对话历史永久保存在数据库，每次都会读取完整历史，就像医生每次都会看你的完整病历。

### Q: 读取历史不是也要时间吗？

**对，但比一直在线节省多了！**

- 读取历史：0.1 秒
- 一直在线：占用 2GB 内存 24 小时
- 就像翻病历 10 秒 vs 医生守在你家门口 24 小时

### Q: 如果我连续问 10 个问题呢？

**还是同样的流程**：

每个问题都创建一个新的 run，每次都读取完整历史（包括前 9 个问题），但每次处理完就关闭会话。

## 类比总结

| 场景 | 接着记忆 | 不接着记忆 |
|---|---|---|
| **医生看病** | 医生一直坐在你家门口等 | 医生看完病就走，下次翻病历 |
| **餐厅** | 厨师只给你一个人做菜 | 厨师做完你的菜，继续做下一桌 |
| **图书馆** | 你一直占着座位看书 | 你离开后别人可以坐，下次拿借阅卡恢复记录 |
| **游戏** | 一直在线挂机 | 退出后存档，下次读档继续 |

## 技术实现

### Pi AgentSession 生命周期

```typescript
// Worker 执行流程
class TaskWorker {
  async executeRun(run: TaskRun): Promise<void> {
    // 1. 从 TaskStore 恢复上下文
    const task = await this.taskStore.getTask(run.taskId);
    const messages = await this.taskStore.getMessages(run.taskId);
    const summary = await this.taskStore.getTaskSummary(run.taskId);
    const workspace = await this.workspaceProvider.materialize(task.workspaceRef);

    // 2. 创建短生命周期 Pi session
    const session = await this.piRunner.createSession({
      task,
      messages,
      summary,
      workspace,
    });

    try {
      // 3. 执行本轮任务
      await session.run();

      // 4. 持久化结果
      await this.taskStore.completeRun(run.id, {
        messages: session.getMessages(),
        events: session.getEvents(),
        artifacts: session.getArtifacts(),
      });
    } finally {
      // 5. 立即销毁 session（无论成功或失败）
      await session.destroy();
    }
  }
}
```

### 关键特性

- **无状态 Control Worker**：Control Worker 不保留任何 Task 的长期状态
- **任意调度**：任何 Control Worker 都可以处理任何 run
- **故障恢复**：Control Worker 崩溃后，另一个 Control Worker 可以从 TaskStore 恢复并继续
- **水平扩展**：可以随时增加或减少 Control Worker 数量

---

## 总结

**一句话**：SiriusX 不让 AI"一直守着你"，而是"做完就走，需要时再回来翻记录"，这样可以同时服务更多人！🚀

**技术视角**：Pi `AgentSession` 是单个 run 的短生命周期执行上下文，不跨节点持久化。多轮连续性来自 TaskStore 中的消息、summary、run 记录、事件日志，以及 workspace 中已经生成和修改过的文件。
