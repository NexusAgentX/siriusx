# 存储模块：生产落盘边界

生产环境采用 **Postgres 记录事实状态，S3 保存文件态和大对象** 的边界设计。所有需要查询、排序、权限判断、恢复决策和审计的数据必须落在 Postgres；所有可能变大、需要下载、需要按 revision 复原的文件内容必须落在 S3 或兼容对象存储。

## 数据分布矩阵

| 数据 | Postgres | S3 / object storage | 说明 |
|---|---|---|---|
| Task metadata | 完整保存 | 不保存 | `tenantId`、`userId`、status、agent/model 锁定字段、workspaceRef |
| Task messages | 保存结构化 parts | 大附件可外置 | 文本消息默认进 PG；附件、图片、大块引用只存 object pointer |
| Task runs | 完整保存 | 不保存 | status、attempt、lease、错误码和时间线都在 PG |
| Run events | 保存可恢复事件 | 大 payload 可外置 | 高频 delta 合并后进 PG；大 stdout/stderr、trace、debug dump 存 S3 |
| Task summary | 保存当前 summary | 历史大版本可外置 | 当前恢复上下文必须能直接从 PG 读取 |
| Artifact index | 保存 path、kind、status、revision、objectKey、etag、size | 保存文件正文 | 页面列表和权限判断走 PG，点击打开时读 S3 |
| Workspace revision | 保存 latest pointer 和 revision metadata | 保存 manifest、file blobs、archive | **两阶段提交**：PG 先创建 pending revision，S3 上传完成后 PG 推进 committed 状态 |
| Sandbox state | Sandbox runtime 进程内存 | 不保存 | 🚨 阶段 0-2 Stateless：无 sandbox_leases 表 |
| Pi debug session/log | 保存索引和 objectKey | 保存 JSONL/log | 只能用于排障，不能作为恢复任务的事实来源 |

## Postgres 表结构

完整的表结构定义在 `schema.sql`。

```text
tasks
  workspace_ref (阶段 0+ workspace://<tenantId>/<taskId>，local/S3 双轨)
  latest_workspace_revision

task_messages
  parts JSONB (MessagePart[])

task_runs
  base_workspace_revision
  committed_workspace_revision
  lease_owner_node_id
  lease_expires_at

task_events
  (task_id, run_id, sequence) UNIQUE
  payload JSONB

task_artifacts
  (task_id, path) PRIMARY KEY
  kind (document|file|code|image)
  status (generated|modified|template|deleted)
  object_key (S3)

workspace_revisions
  status (pending|uploading|committed|failed)
  manifest_object_key (S3)
```

## Events 存储策略

**🚨 Postgres 是事实来源，内存缓冲用于防当前进程丢失**

| 层级 | 存储内容 | 说明 |
|------|----------|------|
| task_events 表 | (task_id, run_id, sequence, type, payload, created_at) | 持久化所有执行事件；`payload` 为 JSONB 列 |
| Worker 内存缓冲 | 同一进程内短暂缓冲，定期批量写入 | 降低 PG 写压力，但不支持跨节点恢复 |

## S3 Key 约定

```text
s3://<bucket>/tenants/<tenantId>/tasks/<taskId>/workspace/revisions/<revision>/manifest.json
s3://<bucket>/tenants/<tenantId>/tasks/<taskId>/workspace/blobs/<sha256>
s3://<bucket>/tenants/<tenantId>/tasks/<taskId>/artifacts/<revision>/<path>
s3://<bucket>/tenants/<tenantId>/tasks/<taskId>/runs/<runId>/tool-output/<sequence>.log
s3://<bucket>/tenants/<tenantId>/tasks/<taskId>/debug/pi-session/<runId>.jsonl
```

## 写入顺序：两阶段提交 + 状态机

1. Worker materialize workspace 到本地 cache。
2. Pi run 修改本地 workspace。
3. **Worker 在 PG 中预创建 workspace revision 记录，状态为 `pending`，设置 `expires_at`（如 5min）**。
4. Worker 上传 changed files / blobs 到 S3，同时更新 revision 状态为 `uploading`。
5. Worker 上传 revision manifest 到 S3，校验 etag/checksum。
6. **Worker 执行 S3 read-after-write check：HEAD 请求验证 manifest 可见性**。
7. Worker 在同一个 PG transaction 中：
   - 更新 revision 状态为 `committed`，记录 `manifest_etag` 和 `committed_at`
   - 推进 `tasks.latest_workspace_revision`
   - 更新 artifacts
   - 完成 run
8. 如果任何步骤失败：
   - Worker 标记 revision 状态为 `failed`
   - `pending`/`uploading` 状态的 revision 由 TTL 过期后自动清理
   - GC 只删除 `status='failed'` 且超过保留期（如 7 天）的 S3 对象

### 关键改进

- **PG 先行**：revision 指针先在 PG 中占位，避免 orphan S3 对象的判断困境
- **明确状态机**：`pending` → `uploading` → `committed` 或 `failed`
- **TTL 自动清理**：pending/uploading 状态超过 `expires_at` 自动标记 `failed`
- **S3 强一致性检查**：上传后立即 HEAD 请求验证可见性，再推进 PG 状态
- **幂等性保证**：Control Worker 崩溃后重试时，可以基于 revision 状态恢复或重新创建

## 读取顺序

- Task snapshot 只读 PG，返回 task、runs、recent messages、summary、artifact index 和 latest workspace revision。
- 文件内容、artifact 下载、大日志和 debug session 通过受控 API 读取 S3；浏览器不直接拼 S3 key。
- Worker 恢复 run 时先从 PG 读取 latest workspace revision，再按 manifest 从 S3 materialize。

## 一致性要求

- `workspace_revisions(revision)` 对同一 task 单调递增或使用内容寻址 revision id。
- **PG 中的 `latest_workspace_revision` 只能引用 `status='committed'` 的 revision**。
- **Worker materialize 时必须检查 revision 状态为 `committed`**，否则等待或失败。
- S3 object 必须记录 checksum/etag；materialize 时校验，失败则本次 run 可重试。
- 删除 Task 不立即硬删 S3；先在 PG 标记删除或完成保留期，再由 GC 删除对象。
- S3 bucket 必须按 tenant/task 前缀隔离，服务端 API 做 auth 和 path guard，不把内部 object key 当作权限边界。

## Storage GC 策略

以下 SQL 示例使用伪代码表达，实际实现需转换为 Unix milliseconds：

- **每小时扫描** `status IN ('pending', 'uploading')` 且 `expires_at < 当前时间` 的 revision，标记为 `failed`
- **每日扫描** `status='failed'` 且 `created_at < 当前时间 - 7天` 的 revision，删除对应 S3 对象
- **每周扫描** S3 对象，删除未被任何 PG revision 引用的 orphan blobs（超过 30 天）

实际 SQL 实现示例：
```sql
-- 扫描过期 revision
SELECT id FROM workspace_revisions
WHERE status IN ('pending', 'uploading')
  AND expires_at < EXTRACT(EPOCH FROM NOW())::bigint * 1000;

-- 扫描超过 7 天的失败 revision
SELECT id FROM workspace_revisions
WHERE status = 'failed'
  AND created_at < EXTRACT(EPOCH FROM NOW())::bigint * 1000 - 604800000;
```

## Auth 和租户边界

生产 SaaS 不能从环境变量硬编码 `userId`。所有用户可见资源都必须从认证上下文派生访问边界。

### 认证模型

- Browser 用户通过平台 session / JWT / OAuth cookie 调用 API。
- 自动化客户端通过 API token 调用 API；token 必须绑定 `userId`、tenant、scope 和过期时间。
- Worker 不代表最终用户登录。Worker 使用内部 service credential 认领 job，但每次执行必须携带 Task 上已经固化的 `userId`、tenant 和权限快照。
- ResultBus、Queue 和 Sandbox service 的内部调用使用 service-to-service credential，不接受浏览器凭据直连。

### 授权规则

- 创建 Task：调用者必须有 `task:create`，Task 的 `userId` 和 tenant 来自 auth context，不接受 body 覆盖。
- 读取 Task：默认只有 Task owner、tenant admin 或显式分享对象可见。
- 发送消息 / 取消 run / close task：默认只有 Task owner 或具备 task write 权限的角色可操作。
- 读取 workspace file / artifact：复用 Task read 权限，并经过 path guard。
- Control Worker claim job：只能通过内部 worker credential，并校验 job 对应 Task 仍存在且未关闭。

### 审计字段

```typescript
interface AuthContext {
  userId: string;
  tenantId: string;
  // 对齐 STAGE0_SPEC §4.1：worker / system 用于后台 sweeper、GC、title/summary job
  actorType: "user" | "api-token" | "worker" | "system";
  scopes: string[];
  tokenId?: string;
}
```

`TaskStore` 中的 task、message、run、cancel、close 操作都应记录 `actorUserId` / `actorType`，至少满足后续审计和误操作追踪。
